# 42 트랙

C/C++ 시스템 기본기에서 컨테이너 운영과 실시간 웹 캡스톤까지 이어지는 공통 기반 트랙입니다.
아래 순서는 공식 일방통행 경로이며 프로젝트를 생략하거나 재배치하려면 트랙 자체를 다시 승인합니다.

```text
linux-admin
→ format-printer
→ signal-message-bus
→ thread-dining
→ small-shell
→ stack-sort
→ stl-container
→ irc-relay-server
→ container-stack
→ pong-pong
```

## 공통 진행 방식

1. `main`에서 source, exercise 계약과 release 문서를 읽고 baseline을 실행합니다.
2. 해당 release의 `learning/<release>`에서 notes → answers → practices 순서로 학습합니다.
3. 문제지가 있으면 답지보다 먼저 구현하고 기준 commit과 diff를 비교합니다.
4. L3 결정만 백지에서 다시 설명·구현하고 L2는 source를 보며, L1은 찾아 읽습니다.
5. 저장소가 선언한 build/test와 실제 smoke를 통과한 뒤 다음 프로젝트로 이동합니다.

각 프로젝트의 완료는 검증된 source-only `main`, 그 commit을 가리키는 annotated release tag와
`learning/<release>` branch를 원격에 게시하고, 원격 fresh clone에서 graph·문서 경계와 해당
프로젝트의 필수 검증을 다시 통과한 상태를 뜻합니다. Baseline이 red이거나 release가 로컬에만
있으면 완료로 계산하지 않습니다.

Ref와 문서 commit 규칙은 [`../commit-policy.md`](../commit-policy.md)와
[`../docs-commit-note.md`](../docs-commit-note.md)를 따릅니다.

## 프로젝트

### 0. `linux-admin`

- 책임: Git/Markdown/commit 규율과 Linux 사용자, SSH, sudo, firewall, service/log 기본
- L3: 권한 상승과 원격 접속 경계, 최소 공개 port, 감사 가능한 sudo 정책
- 완료: 문서 검증과 환경별 운영 체크를 실제로 수행하고 이후 commit이 규율을 자연스럽게 준수

### 1. `format-printer`

- 책임: C 가변인자와 포맷 parser, width/precision/flag, counted write와 정적 library
- L3: flags → width → precision → specifier 소비 순서, padding/부호 우선순위, sticky write error
- 완료: library build와 libc 비교 test가 통과하고 조합 우선순위를 백지에서 설명

### 2. `signal-message-bus`

- 책임: 두 POSIX signal로 byte stream을 전달하는 IPC와 ACK/NACK/timeout
- L3: signal-safe handler, MSB-first 조립, `sigprocmask`/`sigsuspend` race 방지, sender 소유권
- 완료: 실제 server/client smoke를 반복 실행하고 긴 메시지 전송과 중도 종료 sender의 소유권
  회복까지 통과하며, 빠른 ACK를 놓치지 않는 이유를 설명

### 3. `thread-dining`

- 책임: pthread, mutex와 monitor를 이용한 dining philosophers
- L3: 홀짝 fork 획득, state/print mutex 분리, deadline monitor와 종료 조건
- 완료: deadlock·data race 방어를 설명하고 실제 concurrency test를 통과

### 4. `small-shell`

- 책임: lexer → parser → expansion → builtin → fork/pipe/exec를 가진 C shell
- L3: quote/expansion, fd 소유권, pipeline status, parent에서 실행하는 stateful builtin
- 완료: builtin, pipeline, redirect, heredoc과 syntax-error smoke가 통과하고 fd 수명을 설명

### 5. `stack-sort`

- 책임: 제한된 stack 명령으로 정렬하는 C program과 독립 checker/test runner
- L3: 좌표 압축, tiny-sort 분기, LSD radix, 공유 primitive와 emit 경계
- 완료: parser/checker/stress test가 통과하고 값 범위와 무관한 radix 전략을 재구성

### 6. `stl-container`

- 책임: C++98 `vector`, `stack`, `map`, iterator와 allocator 재구현
- L3: 원소 생명주기, SFINAE, iterator traits, red-black tree 균형과 예외 안전
- 완료: 표준 container 행동 대조가 통과하고 self-aliasing과 allocator rollback을 설명

### 7. `irc-relay-server`

- 책임: C++17 BSD socket, kqueue/epoll, IRC framing, outbound queue와 backpressure
- L3: event backend 추상화, partial read/write, protocol/session 경계, heartbeat와 rate limit
- 완료: 실제 TCP smoke가 통과하고 connection·buffer·queue 소유권을 설명

### 8. `container-stack`

- 책임: Docker/Compose 기반 Nginx, PHP-FPM, WordPress, MariaDB와 secret/readiness 체인
- L3: 443 진입 경계, service health dependency, idempotent bootstrap, persistent volume과 secret
- 완료: 정적 config 검사와 가능한 환경의 HTTPS smoke가 통과하고 재기동 안전성을 설명

`container-stack`은 Backend 단계가 아니라 42 트랙의 `pong-pong` 선행입니다. Backend에서는 이
지식을 재사용할 뿐 프로젝트를 중복 배치하지 않습니다.

### 9. `pong-pong`

- 책임: TypeScript, Next.js, Fastify WebSocket, PostgreSQL과 Docker Compose 풀스택 캡스톤
- L3: server-authoritative game state, tick/broadcast, shared contract, repository transaction과 auth
- 완료: workspace test/build, browser smoke와 Compose 기동이 통과하고 서버가 client 입력을
  신뢰하지 않는 경계를 설명

## 트랙 완료

10개 프로젝트의 필수 gate가 통과하고 process/fd, signal, concurrency, allocator, network,
container와 server-authoritative state를 백지에서 연결해 설명할 수 있으면 완료입니다. 이후 Backend와
Frontend로 분기할 수 있으며, 두 트랙은 서로 병렬 가능하지만 각 트랙 내부 순서는 유지합니다.

Red baseline 또는 원격에 게시·fresh-clone 검증되지 않은 release는 위 완료 수에 포함하지 않습니다.

10개 프로젝트 모두 deployable `main`, annotated `codex-5.6` tag와 immutable
`learning/codex-5.6` branch를 원격에 게시하고 fresh-clone gate를 통과했습니다.
`signal-message-bus`도 반복 smoke, 긴 메시지, abandoned-sender 회복과 상속 signal mask
검증을 통과했습니다. 각 release의 알려진 제약과 의도적 비범위는 해당 learning branch의
노트에 보존하며, 이를 구현되지 않은 추가 기능으로 확대하지 않습니다.

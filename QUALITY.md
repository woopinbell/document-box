# QUALITY.md — 품질 검토·개선 오케스트레이터 (read-write)

이 문서를 읽은 세션은 **품질 검토 + 개선 오케스트레이터**로 작동을 시작한다. 목적은 모든 독립 레포의
학습 산출물(루트 `notes/` · `docs/notes/` · `docs/commits/` 답지 · `docs/practice/` 문제지)이 **정의된
역할·품질 기준**을 만족하는지 점검하고, 미달이면 **고치는** 것이다.

> **AUDIT.md와의 분담.** `AUDIT.md` = 읽기전용 **탐지·보고**(고치지 않음). `QUALITY.md` = **탐지 → 개선(수정)
> → 재검증 → 커밋**(read-write). QUALITY 세션은 탐지 단계에서 AUDIT.md §3 체크리스트를 그대로 재사용하고,
> 거기에 §1의 추가 게이트를 더한 뒤, 미달 항목만 §3 절차로 개선한다.
>
> 시작 전 참조: 루트 `CLAUDE.md`(라우터) → `document-box/docs-commit-note.md`(빌드/파생 정본) ·
> `LEARNING.md`(역할 정의 §2) · `commit-policy.md`(커밋·날짜) · `AUDIT.md`(탐지 체크리스트).

## 0. 원칙

1. **스크립트 금지 — 본문을 직접 정독·판단한다.** 검토·탐지·수정을 **임의 스크립트(grep/awk/sed/python
   일괄 등)로 돌리지 않는다.** 스크립트는 숫자·통과여부만 내놓고 *작성된 본문 자체를 읽지 않으므로*, 코드
   충실도·환각·블랭킹·드리프트·문장 품질을 실제로 검토할 수 없다(스크립트를 짜는 순간 본문 검토를 건너뛴
   것이다). 품질 판단은 **반드시 문서를 열어 한 줄씩 읽고** 한다.
   - **허용**: 파일 위치 파악용 `ls`/`find`, 그리고 레포가 **제공하는 검증 명령**(`make test`/`make check-docs`
     등)의 **실행**(그건 본문 검토가 아니라 산출물 검증이라 §5에서 돌린다).
   - **금지**: 본문을 *대신 읽어주는* 분석 스크립트(식별자 grep·fence 카운트·드리프트 diff·NUL 스캔 등)와
     일괄 수정 스크립트(바이트 치환 등). 이상 문자·결함도 **정독으로** 찾고 **에디터로** 고친다.
2. **근거 없는 수정 금지.** 모든 개선의 근거는 `git show`/실소스를 **열어 읽어** 확인한 것이다. 못 찾으면
   멈추고 보고(추측·환각 금지).
3. **최소 개선.** 역할 정의(LEARNING §2)에 맞추는 데 필요한 만큼만 고친다. "다르게 썼겠다"는 결함이 아니다.
4. **한 레포 = 한 단위.** 한 레포를 닫고(탐지→개선→재검증→커밋) 다음으로 간다.
5. **고쳤으면 반드시 재검증.** 수정이 §5 게이트를 통과하는지 확인한 뒤에만 커밋(검증 통과 후에만 커밋).
6. **자가보고 불신.** 서브에이전트·이전 세션의 "했다"를 믿지 말고, 오케스트레이터가 핵심 문서를 **직접
   정독해** 재확인한다(AUDIT §6).

## 1. 품질 기준 — 무엇이 "통과"인가 (역할별 명문 게이트)

LEARNING §2가 역할을 정의하지만 품질 게이트는 dual-form에만 있었다. 아래는 **4개 아티팩트 전부의 통과선**이며,
QUALITY 세션이 강제한다(이게 이 문서가 "추가하는 룰"이다). 판정은 **정독으로** 내린다.

| 아티팩트 | 통과 기준 | 정독 시 확인 지점 |
|---|---|---|
| 루트 `notes/<stack>.md` | **4부 포맷 완비**(Part 1~4 존재 + 각 파트 실질 내용), 스택 1개=1노트, 골격 잔재 없음 | 열어 읽었을 때 Part 1~4가 다 있고 내용이 실질인가; "골격/빌드 세션이 저작" 배너·TODO 잔재가 보이나 |
| `notes/reference-impl/<stack>/` | 노트와 짝, **빌드·실행 가능한 최소체**(핸즈온 레포는 면제 — SEQUENCE 표기 따름) | 디렉터리·README 있나(빌드는 §5 검증에서 시도) |
| `docs/commits/NNN.md` (답지) | **완성 코드 + 이유**(코드 커밋이면 코드블록으로 실제 코드를 보여줌 — **산문-only 금지**, Phase 0″), 실코드 근거(환각 0), `## 검증` 실행가능, 끝에 L3/L2/L1 | 읽었을 때 **실제 코드를 보여주나**(산문 설명만이면 불합격); 그 코드가 `git show`(열어 읽음)와 같나; `## 검증`이 실행가능한가 |
| `docs/practice/NNN.md` (문제지) | 답지에서 파생, 계약 노출/본문 `// TODO 책임`/**책임 줄 누설 금지**, work-unit `###`==답지, 실체성 게이트 | 시그니처는 보이고 본문은 TODO인가; 책임 줄이 답을 누설 안 하나; `###` 제목을 답지와 **눈으로 한 줄씩 대조**(회고 `###` 제외) |
| `docs/notes/<cat>/` | 얕은 온디맨드 어휘(선택적), 루트 notes와 *중복 정독 대상 아님* | 있으면 얕은 어휘인가; 없는 레포는 결함 아님 |
| 교차 일관성 | 허브 문서(SEQUENCE/LEARNING/mobile-track 등)의 사실이 레포 ADR/실소스와 일치 | 약칭·계약을 실소스를 **열어** 대조(예: STOMP 목적지 ↔ gateway 소스) |
| 공통(전 문서) | **이상 제어문자 없음**, commit-policy 준수(영어 명령형 제목+한국어 본문, AI 트레일러 없음) | 렌더 깨짐·안 보이는 문자 흔적 없나; 트레일러 없나 |

**비구현 커밋은 답지만**(마커·순수문서·순수 DTO): 문제지 없음이 정상(실체성 게이트, docs-commit-note §표준절차 3).

## 2. 탐지 (review) — 직접 정독

AUDIT.md §3(구조·실체성·환각·검증·블랭킹·드리프트)을 적용하되 **스크립트로 돌리지 말고 문서를 직접 정독**한다
(§0.1). 레포당:

1. `docs/commits/`·`docs/practice/`·`notes/`의 문서를 **하나씩 열어 읽는다**(목록만 `ls`로 잡는다).
2. **답지**: 실제 코드를 보여주나(산문-only 아닌가) · 그 코드를 `git show <hash>`(역시 **열어 읽어**) 대조해
   충실한가(환각) · `## 검증`이 실행가능한가 · 끝에 L3/L2/L1 있나.
3. **문제지**: 계약 노출/본문 TODO/책임 누설 여부 · work-unit `###`을 답지와 **한 줄씩 눈으로 대조**(드리프트).
4. **노트**: Part 1~4가 실질인가 · 골격 잔재 없나.
5. **교차**: 허브 문서의 약칭·계약을 실소스를 열어 확인.
6. 읽다 안 보이는 제어문자/렌더 깨짐이 의심되는 부분을 표시(작성 단계에서 안 넣는 게 먼저다).

규모가 커서 전수 정독이 버거우면 §4로 **읽기 자체를 분담**한다(스크립트로 대체하는 게 아니라 사람/에이전트가
나눠 읽는다). 판정은 AUDIT §5 형식(PASS/FAIL/ENV-LIMIT/확인필요)에 **개선 계획**을 덧붙인다.

## 3. 개선 (improve) — read-write 절차

미달 유형별로(모두 실소스를 **열어 읽고** 근거 확보):

- **산문-only 답지(Phase 0″)** → docs-commit-note §표준절차 2 Phase 0″: `git show <hash>`의 실코드를 답지에
  **보여주도록** 채운다(시그니처+본문). 기존 프로즈 설계근거·ADR·L3/L2/L1는 '이유'로 유지(삭제 금지).
- **문제지 코드 스켈레톤 미흡/드리프트** → 답지에서 **재파생**(파생 규칙). 손으로 코드 채우지 않는다.
- **노트 4부 미완/골격 잔재** → 4부 본문 저작(개념→본코드분석→미니→심화) + reference-impl 채움.
- **교차 사실 불일치** → 실소스를 ground truth로 허브/문서 정정(예: mobile-track STOMP ↔ gateway 실소스).

**절차(레포 1개):** ① 탐지(§2 정독) → ② **exemplar 1개 직접 수정**(가장 어렵거나 대표 단위 = 판단의 골드
스탠다드) → ③ 나머지 개선(직접 또는 §4 서브에이전트가 **정독해** 미러) → ④ **재검증**(§5) → ⑤ 커밋(§6).

## 4. 서브에이전트 방법 (이 워크플로 전용) ★

이 워크플로는 서브에이전트 사용을 **허용**한다. **단 fan-out의 목적은 "정독(읽기)을 나누는 것"이지 스크립트로
돌리는 게 아니다**(§0.1). 각 에이전트도 자기 범위를 **읽어서** 판단한다.

### 4.1 쓸지 · 어떻게 나눌지 (구획 분배 모델)

- **규모로 결정.** 고칠 파일이 **소수(≲5~6)면 인라인**(서브에이전트 cold-start 비용이 이득보다 큼). 많을 때만 fan-out.
- **구획은 오케스트레이터가 런타임에 정한다**(고정 스킴 없음 — 먼저 §2로 정독·탐지한 뒤 규모를 보고 카브한다). 두 층:
  - **여러 레포** → **레포 1개 = 서브에이전트 1개**(AUDIT §2 · docs-commit-note §작업모델), 동시 **4~6** 배치.
  - **한 레포가 큼(문서 ≳40)** → 그 레포 오케스트레이터가 **연속 NNN 범위(~20개씩)** 로 분할, 동시 **3~4**.
  - 분배는 **disjoint 정적 카브**다 — 오케스트레이터가 범위를 미리 갈라 배치로 던진다(에이전트가 큐에서 집어가는
    동적 work-stealing 아님). 겹침 0 → 쓰기 충돌 0.
- **exemplar는 동질 클러스터(스택/트랙)별 1개.** 같은 스택의 레포 묶음엔 exemplar 하나를 두고 그 에이전트들이
  미러한다 → 묶음 내 일관성. **이종 스택을 한 exemplar로 묶지 말 것** — 원리(계약 노출/본문 비움/산문-only 금지)는
  보편이나 *모양*(시그니처·블랭킹 형태)은 스택마다 달라, 예컨대 Python exemplar가 C++/Java/TS 레포엔 그대로 안
  옮겨간다. **클러스터가 바뀌면 그 스택 대표 레포로 exemplar를 새로 보정**한다.
- 탐지(정독 스윕)·개선(쓰기) 모두 `general-purpose`가 **문서를 읽어** 수행(스크립트 아님, §0.1).

### 4.2 하이브리드 패턴 (검증된 기본형)

1. **오케스트레이터가 exemplar 1건을 직접 완성**한다(가장 어려운/대표 단위). 이게 코드 인라인·파생·블랭킹
   *판단*의 골드 스탠다드다.
2. **`general-purpose` 서브에이전트를 disjoint 연속 범위로 분배**(예: 3개 × 4문서). 범위가 겹치지 않게 나눠
   쓰기 충돌 0.
3. **동시 실행 ≤ 3~4**(집계 품질 한도, AUDIT §2와 동일).
4. 각 서브에이전트 디스패치 프롬프트에 **반드시 포함**:
   - (a) 대상 레포 절대경로 + **담당 NNN 범위**(다른 범위·exemplar 건드리지 말 것).
   - (b) "`document-box/QUALITY.md`(§0 스크립트 금지·§1 기준·§3 절차) + `docs-commit-note.md`(파생 규칙) +
     `commit-policy.md`를 읽고 따르라."
   - (c) **exemplar 포인터**(완성된 답지+문제지 경로) — "판단을 미러하라(모양 베끼기 아님)".
   - (d) **본문을 직접 정독하라 — 스크립트로 검사하지 말 것(§0.1).** ground truth = `git show`/실소스를 열어
     읽기, 추측 금지(답지의 모든 코드 라인은 소스/히스토리에 존재해야 함).
   - (e) **커밋·푸시 금지** — 오케스트레이터가 마지막에 한 번만 커밋(나눠도 커밋은 레포 세션이 한 번).
   - (f) **반환 형식**: 문서별로 *읽고 판단한* 결과(답지가 실제 코드를 보여주나/문제지 블랭킹 적정한가/`###`이
     답지와 같나) + 소스에서 못 찾은 식별자(환각 의심)와 그 근거.
5. **오케스트레이터가 전수 감사**: 서브에이전트 반환을 믿지 말고 표본(특히 L3·가장 어려운 단위)을 **직접
   정독해** 재확인한다(AUDIT §6). 드리프트·잔결함은 직접 교정.
6. **단일 커밋 → (킥오프에서 지시받았으면) 원격 푸시**(§6 · §8).

### 4.3 실패·함정 대비 (실전 교훈)

- **서브에이전트 중도 실패(소켓 드롭 등)**: 결과를 신뢰하지 말고 어디까지 썼는지 **문서를 열어 확인**하고,
  누락분은 오케스트레이터가 직접 마무리한다. 한 번 드롭한 큰 작업은 재개보다 직접 완료가 안전할 때가 많다.
- **제어문자 금지**: 본문 작성 시 NUL(0x00) 같은 **실제 제어문자를 타이핑하지 말 것**. 표기가 필요하면 그
  이스케이프를 **일반 텍스트로** 쓴다(예: STOMP 종결자는 "역슬래시 u 0000" 여섯 글자를 그대로 입력). `Write`
  도구는 넣은 문자를 그대로 기록하므로 의도한 텍스트만 넣으면 된다. **바이트 일괄치환 같은 스크립트 교정은
  금지(§0.1)** — 이상 문자도 정독으로 찾아 에디터로 고친다.
- **per-repo `check-docs`의 한계**: 레포 자체 체커는 구조만 보고 **본문 품질을 다 못 잡는다**(예: 캡스톤
  체커가 "답지 코드 존재"를 미검사해 산문-only가 통과됨). 그래서 §1 기준은 체커가 아니라 **정독**으로 강제한다.

## 5. 재검증 게이트 (한 레포 "개선 완료"의 정의)

수정 후 아래를 모두 만족해야 커밋한다(앞 6개는 **정독으로**, 마지막은 레포 검증 명령 **실행**으로):

- [ ] 답지를 전부 정독 → 전부 **실제 코드를 보여줌**(산문-only 0).
- [ ] 답지 코드를 `git show`와 **읽어 대조** → 환각 없음(다른 이름·since-removed·라이브러리는 근거와 함께 분류).
- [ ] 문제지 work-unit `###`을 답지와 **눈으로 대조** → 드리프트 0; 시그니처 노출·본문 TODO·책임 누설 없음.
- [ ] 블랭킹 표본 1~2개 정독 → 양호.
- [ ] 본문에 안 보이는 제어문자 없음(정독; 작성 시 안 넣음).
- [ ] **레포 검증 명령 실행**: `make check-docs`/`make test`(있으면) → green / ENV-LIMIT 사유 명시 / FAIL이면 미커밋.

## 6. 커밋·푸시 (개선 후)

**정본: [`commit-policy.md`](commit-policy.md).** 요약 — 영어 명령형 제목 `<type>(<scope?>): …` + 한국어
`[근거]/[변경]/[검증]`, **AI 트레일러 없음**, KST·author==committer·근무시간·중복금지.

**최종 종료 단계**(모든 대상 레포가 §5 재검증을 통과한 뒤): 변경 종류별로 **커밋 → 원격이 있으면 푸시**. 이게
한 번의 quality run을 닫는 마지막 행위다. 킥오프(§8)가 "푸시까지"를 지시했을 때 수행하며, 원격이 없으면 로컬
커밋까지만 하고 보고한다.

- 변경 종류로 커밋 분리: 답지 `docs(commits)` / 문제지 `docs(practice)` / 노트 `docs(notes)` / 허브 `docs(<area>)`.
- **바뀐 파일만** 스테이징(`-A` 금지). 서브에이전트로 나눴어도 **커밋은 레포 세션이 마지막에 한 번**.
- **푸시는 원격이 있고 + 지시가 있을 때만**. 원격 없으면 로컬 커밋까지만 하고 보고. 새 원격 생성·공개 전환 등
  outward 행위는 **사용자 확인 후**.

## 7. 범위

QUALITY는 **학습 산출물 문서·노트의 품질**을 다룬다. 앱/서비스 **소스 코드 자체의 리팩터링·기능 변경은
범위 밖**(빌드 세션 몫) — QUALITY는 소스를 ground truth로 **읽기만** 하고, 답지·문제지·노트·허브 문서를
소스에 **맞춘다**. 소스에 실제 결함을 발견하면 고치지 말고 보고한다.

## 8. 킥오프 (이 세션을 여는 법)

새 세션에 아래를 준다(`<대상>`·푸시 여부만 채움). 이 한 덩이가 §0~§7을 작동시킨다:

```
~/Desktop/seungwoo 에서 품질 검토·개선 오케스트레이터로 작동하라.
document-box/QUALITY.md 를 읽고 그대로 따른다(그것이 가리키는 docs-commit-note.md·commit-policy.md·AUDIT.md 포함).
대상: <전 레포 / 특정 레포 목록>.

- 스크립트 금지: grep/awk/python 등으로 본문을 검사하지 말고 **문서를 직접 정독**해 §1 기준으로 판정(§0.1).
  허용은 ls/find 와 레포 자체 make 검증 실행뿐.
- 절차(§3): ① §2 정독 탐지 → ② exemplar 1건 직접 수정(전역 골드 스탠다드) →
  ③ 서브에이전트로 **읽기 분담**(§4: 여러 레포면 레포1개=에이전트1개, 큰 레포면 연속 NNN 범위;
  동시 3~4(레포 단위면 4~6); 에이전트는 git show/실소스를 열어 읽어 정독·수정만 하고 **커밋·푸시 금지**) →
  ④ 오케스트레이터가 표본을 직접 정독해 전수 감사 → ⑤ §5 재검증 게이트.
- 추측 금지: 모든 근거는 git show/실소스를 열어 읽어 확인.
- **최종: 모든 대상이 §5 통과 후 변경 종류별로 커밋(commit-policy: docs(commits)/docs(practice)/docs(notes)/…,
  영어 명령형 제목+한국어 [근거]/[변경]/[검증], AI 트레일러 없음) → 원격이 있으면 푸시.**
- 끝에 레포별 [PASS/수정내역/커밋 해시/푸시 결과]를 보고하라.
```

"푸시까지"를 빼면 로컬 커밋에서 멈춘다(§6). 새 원격 생성·공개 전환 등 outward 행위는 별도 사용자 확인 후.

### 8.1 전 레포(40개) 런 — 하나의 오케스트레이션으로

**"전 레포" = AUDIT §1 열거 = 현재 40개**(`find . -maxdepth 3 -name .git`의 부모 중 `document-box`·
`portfolio-public` 제외, `plan-box`는 비-git이라 자동 제외; `sportsbook/`·`ai-capstone/`은 하위 각 레포).

한 오케스트레이션 세션으로 **가능하다** — 단 *끊김 없는 단일 실행*에 기대지 말고 **체크포인트식**으로 돈다:

- 오케스트레이터는 40개를 **직접 다 읽지 않는다.** 레포 1개 = 서브에이전트 1개로 **동시 4~6 배치** 분배(정독은
  에이전트에 분산), 오케스트레이터는 분배·표본 감사·커밋·푸시만 한다.
- **레포마다 닫는다(§0.3)**: 그 레포가 §5를 통과하면 **즉시 커밋·푸시 → 다음**. 끝까지 모았다가 한 번에 커밋
  금지 — 진행이 커밋으로 남아야 한다.
- **재개 가능**: 세션이 길어지거나 끊기면 **새 오케스트레이션 세션이 이어받는다** — `git`/푸시 상태로 "이미
  quality 커밋·푸시된 레포"를 스킵하고 나머지만. 그래서 **진행 원장**(완료/남은 레포 목록)을 함께 남긴다.
- **클러스터별 보정**: 새 스택 클러스터를 시작할 때마다 **그 클러스터 대표 1개를 먼저 완주**해 *그 클러스터의*
  exemplar·기준을 세운 뒤 같은 클러스터의 나머지를 확장한다(§4.1 — 전역 exemplar 아님). 이종 스택엔 새 exemplar.
- 큰 단일 레포(문서 ≳40)는 그 담당 에이전트가 자기 안에서 연속 NNN 범위로 재분할(§4.1).

> 40개를 더 통제된 단위로 쪼개고 싶으면 **트랜치**로 돌려도 된다(예: 최근 추가분 8개 → 나머지 그룹). 어느 쪽이든
> "하나의 오케스트레이션"은 같고, 차이는 한 런의 크기뿐이다.

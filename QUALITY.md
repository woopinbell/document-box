# QUALITY.md — 품질 검토·개선 오케스트레이터 (read-write)

이 문서를 읽은 세션은 **품질 검토 + 개선 오케스트레이터**로 작동을 시작한다. 목적은 모든 독립 레포의
학습 산출물(루트 `notes/` · `docs/notes/` · `docs/commits/` 답지 · `docs/practice/` 문제지)이 **정의된
역할·품질 기준**을 만족하는지 점검하고, 미달이면 **고치는** 것이다.

> **AUDIT.md와의 분담.** `AUDIT.md` = 읽기전용 **탐지·보고**(고치지 않음). `QUALITY.md` = **탐지 → 개선(수정)
> → 재검증 → 커밋**(read-write). QUALITY 세션은 탐지 단계에서 AUDIT.md §3 체크리스트를 그대로 재사용하고,
> 거기에 §1의 추가 게이트를 더한 뒤, 미달 항목만 §3 절차로 개선한다.
>
> 시작 전 참조: 루트 `CLAUDE.md`(라우터) → `document-box/docs-commit-note.md`(빌드/파생 정본) ·
> `LEARNING.md`(역할 정의 §2) · `commit-policy.md`(커밋·날짜) · `AUDIT.md`(탐지 체크리스트).

## 0. 원칙

1. **근거 없는 수정 금지.** 모든 개선의 근거는 `git show`/실소스다. 못 찾으면 멈추고 보고(추측·환각 금지).
2. **최소 개선.** 역할 정의(LEARNING §2)에 맞추는 데 필요한 만큼만 고친다. 스타일 취향으로 멀쩡한 문서를
   갈아엎지 않는다. "다르게 썼겠다"는 결함이 아니다.
3. **한 레포 = 한 단위.** 한 레포를 닫고(탐지→개선→재검증→커밋) 다음으로 간다.
4. **고쳤으면 반드시 재검증.** 수정이 게이트를 통과하는지 직접 재실행한 뒤에만 커밋. 안 돌려보고 "통과"
   금지(commit-policy: 검증 통과 후에만 커밋).
5. **자가보고 불신.** 서브에이전트·이전 세션의 "했다"를 믿지 말고 핵심 게이트를 오케스트레이터가 재실행한다
   (AUDIT §6 품질 게이트).

## 1. 품질 기준 — 무엇이 "통과"인가 (역할별 명문 게이트)

LEARNING §2가 역할을 정의하지만 품질 게이트는 dual-form에만 있었다. 아래는 **4개 아티팩트 전부의 통과선**이며,
QUALITY 세션이 강제한다(이게 이 문서가 "추가하는 룰"이다).

| 아티팩트 | 통과 기준 | 기계 신호(빠른 탐지) |
|---|---|---|
| 루트 `notes/<stack>.md` | **4부 포맷 완비**(Part 1~4 존재 + 각 파트 실질 내용), 스택 1개=1노트, 골격 배너·TODO 잔재 없음 | `grep -c 'Part [1-4]'`==4; "골격"·"빌드 세션이 저작" 배너 0 |
| `notes/reference-impl/<stack>/` | 노트와 짝, **빌드·실행 가능한 최소체**(또는 핸즈온 레포는 면제 — SEQUENCE 표기 따름) | 디렉터리 존재 + README; 빌드 시도(없으면 ENV-LIMIT) |
| `docs/commits/NNN.md` (답지) | **완성 코드 + 이유**(코드 커밋이면 ```코드블록``` ≥1 — **산문-only 금지**, Phase 0″), 실코드 근거(환각 0), `## 검증` 실행가능, 끝에 L3/L2/L1 | 실체 커밋 답지의 fence==0 → **불합격(산문-only)** |
| `docs/practice/NNN.md` (문제지) | 답지에서 파생, 계약 노출/본문 `// TODO 책임`/**책임 줄 누설 금지**, work-unit `###`==답지, 실체성 게이트 | TODO≥1; work-unit `###` 집합 == 답지(회고 `###` 제외) |
| `docs/notes/<cat>/` | 얕은 온디맨드 어휘(선택적), 카테고리 정리, 루트 notes와 *중복 정독 대상 아님* | 존재 시 README; 없는 레포는 결함 아님 |
| 교차 일관성 | 허브 문서(SEQUENCE/LEARNING/mobile-track 등)의 사실이 레포 ADR/실소스와 일치 | 약칭·계약을 실소스로 대조(예: STOMP 목적지 ↔ gateway) |
| 공통(전 문서) | **NUL 바이트 0**, commit-policy 준수(영어 명령형 제목+한국어 본문, AI 트레일러 없음) | NUL 스캔; 트레일러 grep |

**비구현 커밋은 답지만**(마커·순수문서·순수 DTO): 문제지 없음이 정상(실체성 게이트, docs-commit-note §표준절차 3).

## 2. 탐지 (review) — 도구

AUDIT.md §3(구조·실체성·환각·검증실행·블랭킹·드리프트)을 그대로 돌리고 §1 추가 게이트를 더한다. 검증된 명령:

```sh
R=<repo>
# (a) 산문-only 답지 탐지: 실체 커밋 답지의 코드블록 수(0이면 Phase 0″ 개선 대상)
for f in "$R"/docs/commits/[0-9]*.md; do echo "$(basename $f) fences=$(grep -c '```' "$f")"; done
# (b) 드리프트: practice work-unit ### == commits work-unit ### ('## 기억/설명 Level' 이후 제외)
#     awk '/^## 기억\/설명 Level/{s=1} /^### /&&!s{print}'  로 양쪽 추출해 비교
# (c) 환각: 답지 선언 식별자 → 소스. **bash로 돌릴 것**(zsh는 $var 미분할). 소스 글롭은 *.kt 와 *.kts 둘 다.
#     ids = grep -rhoE '\b(fun|class|object|interface|data class|enum class|sealed (interface|class)) [A-Za-z_][A-Za-z0-9_]*' "$R"/docs/commits/[0-9]*.md | awk '{print $NF}' | sort -u
#     각 id를 git ls-files '*.kt' '*.kts' 에 grep -wF → 커버 ≥95%(미스는 git log -S로 since-removed/.kts/라이브러리인지 분류)
# (d) NUL 스캔(전 .md): python3 -c "import glob;[print(f) for f in glob.glob('$R/**/*.md',recursive=True) if open(f,'rb').read().count(b'\x00')]"
# (e) 레포 자체 게이트: make check-docs (있으면)
```

판정은 AUDIT §5 반환형식(PASS/FAIL/ENV-LIMIT/확인필요)을 쓰되, QUALITY는 FAIL/미달 항목에 **개선 계획**을 덧붙인다.

## 3. 개선 (improve) — read-write 절차

미달 유형별로:

- **산문-only 답지(Phase 0″)** → docs-commit-note §표준절차 2 Phase 0″: `git show <hash>`의 실코드를
  답지에 **보여주도록** 채운다(시그니처+본문). 기존 프로즈 설계근거·ADR·L3/L2/L1는 '이유'로 유지(삭제 금지).
- **문제지 코드 스켈레톤 미흡/드리프트** → 답지에서 **재파생**(파생 규칙). 손으로 코드 채우지 않는다.
- **노트 4부 미완/골격 잔재** → 4부 본문 저작(개념→본코드분석→미니→심화) + reference-impl 채움.
- **교차 사실 불일치** → 실소스를 ground truth로 허브/문서 정정(예: mobile-track STOMP ↔ gateway 실소스).

**절차(레포 1개):** ① 탐지(§2) → ② **exemplar 1개 직접 수정**(가장 어렵거나 대표 단위 = 판단의 골드
스탠다드) → ③ 나머지 개선(직접 또는 §4 서브에이전트, exemplar 미러) → ④ **재검증**(§5) → ⑤ 커밋(§6).

## 4. 서브에이전트 방법 (이 워크플로 전용) ★

이 워크플로는 서브에이전트 사용을 **허용**한다(품질 작업은 fan-out 가능). 단 아래 방식을 지킨다.

### 4.1 쓸지 말지

- **규모로 결정.** 고칠 파일이 **소수(≲5~6)면 인라인**(서브에이전트 cold-start 비용이 이득보다 큼).
  레포 여러 개거나 한 레포 안 문서가 많을 때(≳10) fan-out한다(docs-commit-note §작업모델: 작으면 인라인,
  크면 ~20개씩 분할·동시 3~4).
- 탐지(읽기전용 스윕)는 `Explore`/`general-purpose`, 개선(쓰기)은 `general-purpose`.

### 4.2 하이브리드 패턴 (검증된 기본형)

1. **오케스트레이터가 exemplar 1건을 직접 완성**한다(가장 어려운/대표 단위). 이게 코드 인라인·파생·블랭킹
   *판단*의 골드 스탠다드다.
2. **`general-purpose` 서브에이전트를 disjoint 연속 범위로 분배**(예: 3개 × 4문서). 파일이 겹치지 않게(NNN
   범위로) 나눠 쓰기 충돌 0.
3. **동시 실행 ≤ 3~4.** 빌드/검사 경합 + 집계 품질 때문(AUDIT §2와 동일 한도).
4. 각 서브에이전트 디스패치 프롬프트에 **반드시 포함**:
   - (a) 대상 레포 절대경로 + **담당 NNN 범위**(다른 범위·exemplar 건드리지 말 것).
   - (b) "`document-box/QUALITY.md`(§1 기준·§3 절차) + `docs-commit-note.md`(파생 규칙) + `commit-policy.md`를 읽고 따르라."
   - (c) **exemplar 포인터**(완성된 답지+문제지 경로) — "판단을 미러하라(모양 베끼기 아님)".
   - (d) **ground truth = `git show`/실소스, 추측 금지** — 답지의 모든 코드 라인은 소스/히스토리에 존재해야 함.
   - (e) **커밋·푸시 금지** — 오케스트레이터가 마지막에 한 번만 커밋(docs-commit-note: 나눠도 커밋은 레포 세션이 한 번).
   - (f) **반환 형식**: 문서별 수치(답지 fence 수·문제지 TODO 수·work-unit ### 일치 여부) + 소스에서 못 찾은
     식별자(환각 의심) 목록.
5. **오케스트레이터가 전수 감사**: 서브에이전트 반환을 믿지 말고 §2 핵심 게이트(환각·드리프트·fence·NUL·
   check-docs)를 **직접 재실행**해 일치 확인(AUDIT §6). 드리프트·잔결함은 직접 교정.
6. **단일 커밋**(§6).

### 4.3 실패·함정 대비 (실전 교훈)

- **서브에이전트 중도 실패(소켓 드롭 등)**: 결과를 신뢰하지 말고 **파일 스캔으로 진척을 확인**(어디까지
  썼나), 누락분은 오케스트레이터가 직접 마무리한다. 한 번 드롭한 큰 작업은 재개보다 직접 완료가 안전할 때가 많다.
- **NUL 바이트 금지**: STOMP 종결자 같은 걸 표기할 때 **실제 0x00을 타이핑하지 말고 리터럴 `\u0000`(여섯 글자)**
  로 쓴다. `Write` 도구는 문자를 그대로 기록하고, `Edit` 도구는 `\n`·`\uXXXX`를 이스케이프로 **해석**하므로
  리터럴 텍스트 삽입엔 `Write`(또는 바이트 단위 치환)를 쓴다.
- **zsh vs bash**: 환각 스캔의 `for id in $ids`는 zsh에서 분할이 안 된다 — `bash`로 돌리거나 zsh 배열(`${(@f)...}`)을 쓴다.
- **per-repo check-docs는 보편 기준을 다 안 잡을 수 있다**(예: 캡스톤 체커가 "답지 코드 존재"를 미검사해
  산문-only가 통과됨). QUALITY는 §1의 보편 게이트를 레포 체커와 **무관하게** 강제한다.

## 5. 재검증 게이트 (한 레포 "개선 완료"의 정의)

수정 후 아래를 모두 통과해야 커밋한다:

- [ ] `make check-docs`(있으면) PASS.
- [ ] 실체 커밋 답지 전부 코드블록 ≥1(산문-only 0).
- [ ] 환각: 답지 식별자 → 소스(`*.kt`+`*.kts`) 커버 ≥95%(미스는 §2(c)로 since-removed/.kts/라이브러리 분류 = 환각 0).
- [ ] 드리프트 0(문제지 work-unit `###` == 답지).
- [ ] 블랭킹 표본 1~2개 양호(계약 노출·본문 비움·책임 누설 없음).
- [ ] 문서 내 NUL 0.
- [ ] 검증 명령 실제 실행(green / ENV-LIMIT 사유 명시 / FAIL이면 미커밋).

## 6. 커밋·푸시 (개선 후)

**정본: [`commit-policy.md`](commit-policy.md).** 요약 — 영어 명령형 제목 `<type>(<scope?>): …` + 한국어
`[근거]/[변경]/[검증]`, **AI 트레일러 없음**, KST·author==committer·근무시간·중복금지.

- 변경 종류로 커밋 분리: 답지 `docs(commits)` / 문제지 `docs(practice)` / 노트 `docs(notes)` / 허브 `docs(<area>)`.
- **바뀐 파일만** 스테이징(`-A` 금지). 서브에이전트로 나눴어도 **커밋은 레포 세션이 마지막에 한 번**.
- **푸시는 원격이 있고 + 지시가 있을 때만**. 원격 없으면 로컬 커밋까지만 하고 보고(commit-policy). 새 원격
  생성·공개 전환 등 outward 행위는 **사용자 확인 후**.

## 7. 범위

QUALITY는 **학습 산출물 문서·노트의 품질**을 다룬다. 앱/서비스 **소스 코드 자체의 리팩터링·기능 변경은
범위 밖**(그건 빌드 세션 몫) — QUALITY는 소스를 ground truth로 **읽기만** 하고, 답지·문제지·노트·허브
문서를 소스에 **맞춘다**. 소스에 실제 결함을 발견하면 고치지 말고 보고한다.
